# Palace Layout — Filesystem Reference

This is the directory structure `mempalace-search` is navigating. Read this file if you need to double-check where things live or what metadata is on each file.

## Top-level tree

```
~/.mempalace/
├── config.yaml              # palace config: mode, palace_path, wings list
├── identity.md              # L0 — always-loaded identity (~100 tokens)
├── critical_facts.md        # L1 — essential story, enriched after mining
├── entities.md              # people, projects, aliases — use for name expansion
└── palace/
    └── <wing>/              # top-level category (project/person/life-area)
        ├── _wing.md         # wing metadata (description, created_at)
        └── <room>/          # sub-category, created lazily by mining
            ├── _room.md     # room metadata (description, keywords)
            └── drawers/
                └── <id>.md  # verbatim drawer files
```

## File types

### `config.yaml`

Top-level palace config written by `mempalace-init`:

```yaml
mode: combo
palace_path: ~/.mempalace/palace
wings:
  - family
  - work
  - health
  - projects
created_at: 2026-04-10T12:00:00Z
```

The `palace_path` may be different from `~/.mempalace/palace` if the user set up a palace in a custom location. Always read `config.yaml` first to find the actual palace root.

### `identity.md`

L0 layer from the upstream `layers.py`. Always loaded when a session "wakes up". Human-edited free-form markdown. Short — ~100 tokens.

### `critical_facts.md`

L1 layer. Auto-generated by init, enriched by mining. Contains:

- People (personal + work, with AAAK-style short codes in parens)
- Projects
- Wings summary

### `entities.md`

Entity registry for name disambiguation. Expand aliases here before searching:

```markdown
## People
- Riley [personal] — daughter
- Devon [personal] — partner (alias: D)

## Aliases
D → Devon
```

When the user searches for "D", expand to search for both "D" (as a name) and "Devon".

### `_wing.md` and `_room.md`

Metadata files at each wing/room level. Skip these when searching drawer content (they're scaffolding, not memories). Identify them by the leading underscore in the filename.

### Drawer files

The actual verbatim memories. Filename is `<drawer_id>.md` where `drawer_id = drawer_<wing>_<room>_<hash24>`. Structure:

```markdown
---
id: drawer_projects_decisions_a1b2c3d4e5f67890abcd1234
wing: projects
room: decisions
source_file: /home/user/code/myapp/ARCHITECTURE.md
chunk_index: 5
added_by: mempalace-mine
filed_at: 2026-04-10T12:34:56Z
source_mtime: 1712745296.0
ingest_mode: project
---

<verbatim chunk content>
```

The frontmatter is always in the first ~10 lines. Parse it before reading the body when triaging candidates — you can eliminate irrelevant drawers without reading their full content.

### Metadata fields reference

| Field | Purpose |
|-------|---------|
| `id` | Stable drawer identifier (matches the filename stem) |
| `wing` | Top-level category |
| `room` | Sub-category |
| `source_file` | Absolute path to the original file this chunk came from |
| `chunk_index` | 0-based index within the source file (for multi-chunk files) |
| `added_by` | Which skill/agent filed this drawer |
| `filed_at` | ISO 8601 timestamp of when the drawer was written |
| `source_mtime` | Unix mtime of the source file when it was mined (for change detection) |
| `ingest_mode` | `project`, `convos`, or `general` |
| `extract_mode` | Only set in general mode — the memory type (decision/problem/milestone/etc.) |
| `memory_type` | Only set in general mode — same as room name for general drawers |

## Reading the frontmatter efficiently

When ripgrep finds a matching file, you don't need to read the whole thing to judge relevance:

```bash
# Read just the frontmatter
head -20 ~/.mempalace/palace/projects/decisions/drawers/drawer_...md
```

The YAML frontmatter is delimited by `---` lines. Everything between the two delimiters is the metadata; everything after the second `---` is the verbatim content.

## Enumerating drawers

- **List all wings**: `ls -1 ~/.mempalace/palace/`
- **List rooms in a wing**: `ls -1 ~/.mempalace/palace/<wing>/ | grep -v '^_'` (strips `_wing.md`)
- **List drawers in a room**: `ls -1 ~/.mempalace/palace/<wing>/<room>/drawers/`
- **Count drawers in a wing**: `find ~/.mempalace/palace/<wing>/ -name "drawer_*.md" | wc -l`

## Empty states

- **No palace**: `~/.mempalace/config.yaml` doesn't exist → user needs to run init.
- **Palace exists, no wings**: `~/.mempalace/palace/` is empty → user needs to mine first.
- **Wing exists, no rooms**: only `_wing.md` is present → no rooms have been created yet (mining creates rooms on demand).
- **Room exists, no drawers**: `_room.md` is present but `drawers/` is empty or missing → something was filed here but then removed, or the room was pre-created.

Treat these four states as distinct when reporting "nothing found".
